## 1. Phase 2 "Keep original ranking" creates ambiguous blanks

**Status:** `[ ]`

**Observed:** Reviewers who selected "Keep original ranking" in Phase 2 show
empty Phase 2 columns in the admin aggregation. Indistinguishable from
reviewers who didn't do Phase 2 at all.

**Proposed direction:** Remove the "Keep original ranking" button so blanks
in Phase 2 unambiguously mean Phase 2 wasn't completed; any submitted Phase 2
has explicit values.

**Notes:**
- Current data model preserves the intent via `phase2_skipped: true`, but the
  aggregation doesn't surface this. Two valid paths:
  - **(a) Remove the button** (proposed) — simplest, cleanest data model going
    forward, but loses the ability to record "I deliberately reviewed Phase 2
    and kept my Phase 1 ranks." For some studies that distinction matters.
  - **(b) Keep the button but display it differently in the aggregation** —
    e.g., show "Same as P1" or copy the Phase 1 values into the Phase 2
    columns when `phase2_skipped` is true. Viewer UX stays intuitive; admin
    view becomes unambiguous.
- Worth a quick team decision before implementation. If (a), existing rankings
  with `phase2_skipped: true` need handling (migrate or accept mixed history).

**Type:** UX + data-model change. **Scope:** small.

## 4. Dose overlay default Upper = plan max (not Rx); slider max also = plan max

**Status:** `[ ]`

**Observed:** Upper currently defaults to the prescription dose. For SRS
especially (where dose far exceeds Rx in the hotspot), this clips
visualization of the hot region.

**Proposed direction:** Default Upper to the plan's maximum dose. Set the
slider's max value to the same.

**Notes:**
- Max dose across plans is already computed at load time (`doseMaxGy` or
  similar in viewer state). Wire it into the slider defaults and the `max`
  attribute.
- Per-plan max varies across A/B/C. Decide: max-across-all-plans (consistent
  comparison) vs per-plan slider state.

**Type:** UX / default change. **Scope:** small.

## 6. Dose distributions: grainy + offset on SRS

**Status:** `[ ]`

**Observed:** Two related issues:
- **Graininess:** SRS dose grids appear blocky when overlaid on CT, making
  conformity hard to assess.
- **Offset:** Dose appears slightly shifted relative to contours on SRS plans.

**Notes:**
- **Graininess** = nearest-neighbor sampling. Solvable by bilinear
  interpolation when rendering the dose grid onto the CT canvas. Modest
  implementation work; real visual improvement.
- **Offset** = likely a coordinate-system issue. SRS dose grids often have:
  - Higher resolution (smaller voxel spacing) than the CT
  - A different origin (often cropped to the target region, with its own
    `ImagePositionPatient`)
  - Different orientation in some cases
- If the rendering code assumes dose grid shares the CT's origin/spacing,
  a small but visible offset on SRS would result. The dose-binary loading
  path needs to use the dose's own `ImagePositionPatient` and `PixelSpacing`
  for coordinate transforms, not the CT's.
- The most clinically meaningful of the visual issues. **A grainy-but-aligned
  dose is bearable; an offset dose is misleading.**

**Type:** bug (offset) + visual polish (graininess). **Scope:** medium (offset
fix touches coordinate math).

## 7. Admin tool workflow + structure colors

**Status:** `[ ]`

### 7a. Move prescription step BEFORE structure mapping / goal evaluation

- Workflow rationale: the Rx is the foundation; relative goals can't be
  evaluated without it; entering it first feels natural.
- Currently Step 5. Would become Step 2 or 3.
- Also fixes the timing wrinkle from the V100% fix — relative goals
  (V100%, D99% > 115%) become evaluable on first pass without backtracking
  to Step 5.

**Scope:** small-medium (HTML reordering + adjusting code that assumes step
ordering).

### 7b. Template structure colors → viewer

- Template XML defines structure colors. Currently the viewer assigns its
  own colors, and identical-colored structures can appear (confusing during
  review).
- Need to flow color from `templateStructures` → `App.contourMappings` (or
  parallel structure) → `manifest.json` → viewer's structure list.

**Scope:** small-medium (data plumbing through the pipeline).

### 7c. Optional palette to override colors in admin tool

- Lets the admin pick distinct colors when the template doesn't provide them,
  or when two important structures share a similar color.
- Small UI affordance per structure mapping row.

**Scope:** small. Lower priority than 7b.
