## 1. Phase 2 "Keep original ranking" creates ambiguous blanks

**Status:** `[ ]`

**Observed:** Reviewers who selected "Keep original ranking" in Phase 2 show
empty Phase 2 columns in the admin aggregation. Indistinguishable from
reviewers who didn't do Phase 2 at all.

**Proposed direction:** Remove the "Keep original ranking" button so blanks
in Phase 2 unambiguously mean Phase 2 wasn't completed; any submitted Phase 2
has explicit values.

**Notes:**
- Current data model preserves the intent via `phase2_skipped: true`, but the
  aggregation doesn't surface this. Two valid paths:
  - **(a) Remove the button** (proposed) — simplest, cleanest data model going
    forward, but loses the ability to record "I deliberately reviewed Phase 2
    and kept my Phase 1 ranks." For some studies that distinction matters.
  - **(b) Keep the button but display it differently in the aggregation** —
    e.g., show "Same as P1" or copy the Phase 1 values into the Phase 2
    columns when `phase2_skipped` is true. Viewer UX stays intuitive; admin
    view becomes unambiguous.
- Worth a quick team decision before implementation. If (a), existing rankings
  with `phase2_skipped: true` need handling (migrate or accept mixed history).

**Type:** UX + data-model change. **Scope:** small.

## 6. Dose distributions: grainy + offset on SRS

**Status:** `[ ]`

**Observed:** Two related issues:
- **Graininess:** SRS dose grids appear blocky when overlaid on CT, making
  conformity hard to assess.
- **Offset:** Dose appears slightly shifted relative to contours on SRS plans.

**Notes:**
- **Graininess** = nearest-neighbor sampling. Solvable by bilinear
  interpolation when rendering the dose grid onto the CT canvas. Modest
  implementation work; real visual improvement.
- **Offset** = likely a coordinate-system issue. SRS dose grids often have:
  - Higher resolution (smaller voxel spacing) than the CT
  - A different origin (often cropped to the target region, with its own
    `ImagePositionPatient`)
  - Different orientation in some cases
- If the rendering code assumes dose grid shares the CT's origin/spacing,
  a small but visible offset on SRS would result. The dose-binary loading
  path needs to use the dose's own `ImagePositionPatient` and `PixelSpacing`
  for coordinate transforms, not the CT's.
- The most clinically meaningful of the visual issues. **A grainy-but-aligned
  dose is bearable; an offset dose is misleading.**

**Type:** bug (offset) + visual polish (graininess). **Scope:** medium (offset
fix touches coordinate math).

## 7. Admin tool workflow + structure colors

**Status:** `[ ]`

### 7a. Move prescription step BEFORE structure mapping / goal evaluation

- Workflow rationale: the Rx is the foundation; relative goals can't be
  evaluated without it; entering it first feels natural.
- Currently Step 5. Would become Step 2 or 3.
- Also fixes the timing wrinkle from the V100% fix — relative goals
  (V100%, D99% > 115%) become evaluable on first pass without backtracking
  to Step 5.

**Scope:** small-medium (HTML reordering + adjusting code that assumes step
ordering).

### 7b. Template structure colors → viewer

- Template XML defines structure colors. Currently the viewer assigns its
  own colors, and identical-colored structures can appear (confusing during
  review).
- Need to flow color from `templateStructures` → `App.contourMappings` (or
  parallel structure) → `manifest.json` → viewer's structure list.

**Scope:** small-medium (data plumbing through the pipeline).

### 7c. Optional palette to override colors in admin tool

- Lets the admin pick distinct colors when the template doesn't provide them,
  or when two important structures share a similar color.
- Small UI affordance per structure mapping row.

**Scope:** small. Lower priority than 7b.

## 8. Some users report Phase 2 submit button not clickable

**Status:** `[ ]`  *(investigate first — likely user error, but verify code)*

**Observed:** Two reviewers reported the Submit button for Phase 2 rankings
appeared grayed out / not clickable, even though (they said) they hadn't
clicked submit.

The workspace correctly grays out after a successful Phase 2 submission, so a
previously-submitted Phase 2 would look "not clickable" to a returning user.

**But:** Want to rule out a real bug in the Phase 2 validation that could keep
the button disabled when fields appear populated.

**Notes:**
- Code review needed: `validateRanking()` inside `initRanking()` in `viewer.js`.
  Currently disables submit unless ALL of: reviewer set, all N ranks selected,
  all N Likerts selected, at least one rank = 1st, ties confirmed if present.
- Worth checking: does Phase 2 handle the case where a user opens a
  previously-submitted-and-skipped ranking? Or a previously-completed Phase 2?
- Definitive test: check the saved JSON in `PROJECTS/_rankings/` for these
  reviewers. If Phase 2 fields are populated → they submitted accidentally.
  If empty → real bug, dig deeper.
- Worth checking: any race between the workspace re-render and the validation
  re-running.

**Type:** investigate first; possibly bug, possibly user error. **Scope:** small.

## 9. Estimated delivery time deviates from TPS-reported values

**Status:** `[ ]`  *(blocked on user-provided test cases)*

**Observed:** During development, the delivery time calculation matched the
TPS-reported value on the original test case. Subsequent plans show
meaningful deviations.

**Notes:**
- Calculation considers MU/dose rate, gantry rotation speed, MLC speed. The
  "Speed-limiting factor" breakdown shows mu_pct / gantry_pct / mlc_pct.
- Different machine models have different rate limits (Halcyon vs TrueBeam vs
  others). The calc may assume one machine's constraints; refining for
  multiple machines would improve fidelity.
- Affects the plan-detail reveal accuracy — delivery time is one of the few
  details shown to reviewers post-Phase 1, so it should be trustworthy.
- **Blocked on:** User to provide a couple of plan files with known-good TPS
  delivery times for calibration / refinement.

**Type:** processing-logic refinement (server side / admin tool). **Scope:**
medium (depends on what we discover from test data).

## 10. New feature work proposed

**Status:** `[ ]`

### 10a. Dose profile tool

- Discussed previously in this project.
- User draws a line on any view (axial / coronal / sagittal).
- A pop-up shows the dose along that line for ALL currently-displayed plans,
  plotted as a profile.
- Useful for comparing penumbra, conformity, and hot regions across plans.
- **Scope:** substantive feature; ~1–2 days of focused work.

### 10b. Rotate views (oblique MPR)

- Discussed at length previously. Two implementation paths:
  - **Server-side resampling** (the pragmatic middle ground I sketched earlier)
    — user defines an oblique plane via crosshair angles; server computes the
    resampled slice and returns it. Works with the current Canvas-2D
    architecture, no client-side performance concerns.
  - **WebGL with 3D textures** — fully interactive rotation, smooth, but
    requires a non-trivial architectural change to the renderer.
- For the current user base, server-side resampling is probably sufficient.
- **Scope:** days (server-side path) vs weeks (WebGL path).

### 10c. Window/level presets

- Dropdown with common CT presets: Soft Tissue (W:400 L:40), Brain (W:80 L:40),
  Bone (W:1500 L:300), Lung (W:1500 L:-600), etc.
- Optional extension: let users save custom presets per project.
- **Scope:** small UI feature; ~1–2 hours.

## 11. Dose-difference comparison between pairs of plans  *(low priority)*

**Status:** `[ ]`  *(low priority per request; depends on #6)*

**Observed:** A colleague asked about visualizing dose differences between
pairs of plans.

**Proposed direction:** Pop-out window where the user selects two plans (e.g.
A vs B); displays a difference map (`doseA - doseB`) with a diverging colormap
(e.g. red where A > B, blue where A < B).

**Notes:**
- **Requires correct dose-grid coordinate handling.** Dose grids may sit on
  different origins / spacings (especially in SRS — see #6). The grids need
  to be resampled onto a common space before subtraction; otherwise the
  difference map would show coordinate-system artifacts as if they were real
  dose differences.
- **#6 should land first** — otherwise the difference visualization would
  amplify whatever offset/sampling issue exists today.
- New pop-out window infrastructure similar to the existing clinical-goals
  popout, with a diverging colormap added to the renderer.
- Flagged as low priority by requester.

**Type:** feature. **Scope:** medium-large.
**Dependency:** #6 (dose coordinate system).
